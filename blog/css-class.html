Best practices

What are the best practices associated with using classes vs. ids?

When to use IDs:

IDs are reserved for unique elements on the page. As such are ideally given names that imply uniqueness, such as navbar or firstpost.

IDs have a special browser function - they can be linked to in order to directly to allow you to navigate directly to that part of the webpage. For example:

<a href = "#thirdpost">

It is easier to access a particular element with the getElementById javascript function than by using classes, so if you plan to manipulate an element with javascript, it should have an ID.

When to use classes

Classes can apply to multiple elements. If another element has the same class (or if there might be another element with one in the future) you should use a class. This will make the code more manageable in the future.

You can apply multiple classes to an element, but not multiple IDs. This can be helpful for streamlining code, but can also make code more difficult to understand, manage and maintain in future.

You can use both

What if you need javascript access to an object, but it also shares styling with multiple objects? No problem, you can give it both a class and an ID.

Disagreements about best practices are on the use of classes ßand IDs abound, but consensus is absolute on at least one issue: Do not give CSS classes names related to styling (like red-box or left-side-bar) Why? To avoid losing the maintenance advantages of CSS. Obviously. Well this may be obvious to a developer with even a few weeks experience, but if you are a total newbie like me with just a few hours under their belt, it may not be so obvious. Here, I’ll take you through it.

A great benefit of having a css stylesheet seperate and independent of your html is that you can make changes to the appearance of your page with minimal or no contact with the html.

Imagine if you were holding a party for your eleventy-first birthday, and you wanted your boring white house to take on a new colour to celebrate the event. You could paint your house a bright new colour scheme, or you could simply project light onto your house and give it any colour you wanted. In fact, once you had set up the lights properly, you could give your house a new colour every night of the year with minimal effort!

Consider the following royalty free image from a building lit in such a way for the Eindhoven GLOW festival.

CSS set up correctly is like such a lighting system.

If you have written your CSS nicely, especially if you have made sensible use of variables, you may be able to change the colour scheme of your whole website by just tweaking a few lines of CSS. This is the ideal we are striving for when we write our CSS. However, if we use stylistic names for our CSS classes, for example if we have classes called “blue-box” and “blue-bar” instead of something like “comment-box” or “side-bar” what happens if we want to change those boxes from blue to yellow? Instead of simply changing the colour in the css, we will have to also go through our HTML and change every occurrence of ‘blue-box’ and ‘blue-bar’ to ‘yellow-box’ and ‘yellow-bar’.

ß

Much better to give the css classes functional or generic names, and keep the styling information independent of the html.
